# Local Frame Operations

Specification for the three new instructions needed to support `letrec` and
local variable mutation (`set!` on non-global bindings).

---

## New VM field: `frame_base`

```cpp
size_t frame_base = 0; // added to Stack
```

A single scalar on the VM â€” not a stack. It records the index into `data_stack`
where the current function's local frame begins. `ENTER` sets it; no other
instruction touches it. Because it is a scalar, a callee's `ENTER` overwrites
the caller's value. This is intentional: `SET_LOCAL` is only valid in a
function's prologue, before any `CALL` is made.

---

## ENTER `<n>`

Operand kind: `U64`
Operation kind: `CONTROL`
Stack effect: none (0 pops, 0 pushes)

Sets `frame_base` to the index of the first local in `data_stack`:

```
frame_base = data_stack.size() - n
```

Must be the first instruction in any function body that uses local variables.
`n` is the number of locals the frame owns â€” equal to the number of formals
passed as arguments by the caller (not counting captured variables, which land
above the formals after `CALL` restores them).

```
before: ... loc_0  loc_1  ...  loc_{n-1}      (loc_{n-1} on top)
after:  ... loc_0  loc_1  ...  loc_{n-1}      frame_base = &loc_0
```

Generator responsibility: emit `ENTER n` where `n = formals.size()` at the
start of each lambda body that contains `set!` on a formal or a `letrec`
binding. Track `frame_base` as a compile-time offset equal to
`current_stack_depth - n` at that point in the emitted bytecode.

---

## GET_LOCAL `<i>`

Operand kind: `U64`
Operation kind: `CONTROL`
Stack effect: 0 pops, 1 push

Clones the cell at `data_stack[frame_base + i]` and pushes it.

```
before: ... loc_0  ...  loc_i  ...
after:  ... loc_0  ...  loc_i  ...  clone(loc_i)
```

Equivalent to `PICK` with a depth computed from the generator's knowledge of
the current stack depth relative to `frame_base`. The generator may emit `PICK`
instead of `GET_LOCAL` when it can statically determine the correct depth â€” in
particular after any `CALL` where `frame_base` has been clobbered. `GET_LOCAL`
is only safe before the first `CALL` in a function body or after a re-issued
`ENTER`.

---

## SET_LOCAL `<i>`

Operand kind: `U64`
Operation kind: `CONTROL`
Stack effect: 1 pop, 0 pushes

Pops the top of `data_stack` and writes it into slot `i` of the current frame:

```
before: ... loc_0  ...  loc_i  ...  value
after:  ... loc_0  ...  value  ...           (loc_i replaced, stack shrinks by 1)
```

Implementation:

```cpp
auto value = std::move(data_stack.back());
data_stack.pop_back();
data_stack[frame_base + i] = std::move(value);
```

`SET_LOCAL` must only appear in the function prologue â€” before any `CALL` â€”
because a callee's `ENTER` will overwrite `frame_base`.

---

## ISA additions

In `isa.hpp`, append to `Operation`:

```cpp
ENTER, GET_LOCAL, SET_LOCAL,
```

Update `op_count` and append to `spec_list`:

```cpp
{"enter", OperandKind::U64, OperationKind::CONTROL, 0, 0},
    {"get_local", OperandKind::U64, OperationKind::CONTROL, 0, 1},
    {"set_local", OperandKind::U64, OperationKind::CONTROL, 1, 0},
```

## Worked example: letrec

Source:
```scheme
(letrec ((even? (lambda (n) (if n (odd? (- n 1)) 1)))
         (odd?  (lambda (n) (if n (even? (- n 1)) 0))))
  (even? 4))
```

After desugaring this becomes:
```scheme
((lambda (even? odd?)
   (set! even? (lambda (n) ...))
   (set! odd?  (lambda (n) ...))
   (even? 4))
 undef undef)
```

Generated bytecode outline for the outer lambda body:

```
; stack on entry: [ undef(even?)  undef(odd?) ]
ENTER 2              ; frame_base = 0,  locals: [even?=0, odd?=1]

; build closure for even? (captures odd? â€” slot 1 â€” via GET_LOCAL or PICK)
...
MKCLOSURE <even?_code>
SET_LOCAL 0          ; data_stack[0] = even?_closure,  undef replaced

; build closure for odd? (captures even? â€” slot 0)
...
MKCLOSURE <odd?_code>
SET_LOCAL 1          ; data_stack[1] = odd?_closure,   undef replaced

; body: (even? 4)
; frame_base is still valid here â€” no CALL has happened yet
GET_LOCAL 0          ; push even?_closure
PUSH 4
CALL
```

After `CALL`, `frame_base` belongs to the callee. The outer lambda does not
use `GET_LOCAL` or `SET_LOCAL` again, so no conflict arises.

---

## Invariants

- `frame_base` is only meaningful after `ENTER` has been executed in the
  current function.
- `SET_LOCAL i` must be emitted only before the first `CALL` in a function
  body (i.e., in the prologue).
- `GET_LOCAL i` is safe before the first `CALL`. After a `CALL`, the generator
  should use `PICK` with a statically computed depth, or re-emit `ENTER` to
  reestablish `frame_base`.
- `i` must satisfy `frame_base + i < data_stack.size()` at the point of
  execution.

# Calling Convention

---

## Roles

- **Caller** â€” the code that sets up and initiates a call.
- **Callee** â€” the lambda body that executes after `CALL` jumps to `code_idx`.

All functions are closures. There are no "bare" function pointers â€” every call
goes through a closure handle produced by `MKCLOSURE` (or loaded via
`LOADGLOBAL` for top-level defines).

---

## Stack layout at function entry

After `CALL` completes and before the callee's first instruction:

```
[ ... caller frame ... | arg_0 | arg_1 | ... | arg_{k-1} | cap_0 | cap_1 | ... | cap_{m-1} ]
                        ^                                                                    ^top
```

- **args** (`k` of them): pushed left-to-right by the caller before the closure
  handle. `arg_0` was pushed first and sits deepest in the callee's region.
- **captures** (`m` of them): restored by `CALL` from the heap env in their
  original left-to-right order. They land on top of the args.
- The closure handle itself is consumed by `CALL` and does not appear.

After `ENTER (k + m)`:

```
frame_base = data_stack.size() - (k + m)   â†’   points at arg_0

GET_LOCAL 0       .. GET_LOCAL k-1    â†’  arg_0 .. arg_{k-1}
GET_LOCAL k       .. GET_LOCAL k+m-1  â†’  cap_0 .. cap_{m-1}
```

`n` in `ENTER n` is therefore `formals.size() + captures.size()`, both of which
are statically known to the generator at the point of emitting the lambda.

---

## Caller contract

1. Evaluate and push arguments **left-to-right**.
2. Push the closure handle (via `GET_LOCAL`, `PICK`, `LOADGLOBAL`, or as the
   result of `MKCLOSURE`).
3. Execute `CALL`.
4. After `RET`, the top of `data_stack` holds the callee's single return value.
   The callee has already removed its own frame â€” the caller's stack is exactly
   as it was before step 1, plus the return value on top.

---

## Callee contract

1. Execute `ENTER (formals + captures)` as the first instruction â€” establishes
   `frame_base`.
2. **Prologue** (optional): emit `SET_LOCAL` to initialise mutable bindings
   (e.g. `letrec`). No `CALL` may appear between `ENTER` and the last
   `SET_LOCAL`.
3. **Body**: evaluate the expression. `GET_LOCAL` is valid here; after any
   `CALL` in the body, use `PICK` with a generator-computed depth instead.
4. **Epilogue** (frame cleanup): the return value is on top of the frame slots.
   Sink it below them and drop the frame before `RET`:

   ```
   ; n = formals + captures
   NROT (n + 1)    ; move return value below all n frame slots
   DROP            ; Ă— n   (one per frame slot)
   RET
   ```

   For `n = 1` this reduces to `SWAP DROP RET`.
   For `n = 0` (no formals, no captures) this reduces to `RET` alone.

After `RET`, `data_stack` contains the caller's original items plus the return
value. `frame_base` is stale â€” any subsequent use by the caller must go through
`PICK`.

---

## Regular (non-recursive) call

```
Caller emits:
  <evaluate arg_0>        ; push arg_0
  ...
  <evaluate arg_{k-1}>    ; push arg_{k-1}
  <load closure handle>   ; push handle
  CALL

Callee body starts at code_idx:
  ENTER (k + m)           ; frame_base â†’ arg_0
  <prologue: SET_LOCAL â€¦>
  <body>
  NROT (k+m+1)
  DROP Ă— (k+m)
  RET
```

Stack state at key moments:

```
before CALL:   ... | arg_0 | ... | arg_{k-1} | handle
after  CALL:   ... | arg_0 | ... | arg_{k-1} | cap_0 | ... | cap_{m-1}
after  ENTER:  (same, frame_base set)
after  body:   ... | arg_0 | ... | cap_{m-1} | result
after  NROT:   ... | result | arg_0 | ... | cap_{m-1}
after  DROPs:  ... | result
after  RET:    (caller resumes, result on top)
```

---

## Recursive call

A recursive call is structurally identical to a regular call. The closure
handle for the recursive callee is obtained by `GET_LOCAL` (if the binding is
in the current frame) or `PICK` (if the call occurs after other `CALL`s have
clobbered `frame_base`).

The only difference from the caller's perspective: the recursive invocation
will execute `ENTER` and overwrite `frame_base`. This is safe because:

- `SET_LOCAL` only appears in the prologue, which completes before any `CALL`.
- After a recursive `CALL` returns, the outer frame's `frame_base` value is
  stale â€” but the outer frame is also past its prologue, so it no longer needs
  `frame_base`.

Concrete example â€” `even?` calling `odd?` which calls `even?`:

```
; inside even? body, after prologue
; frame_base points to even?'s frame (n, odd?-capture)
GET_LOCAL 1      ; push odd?-closure
<evaluate arg>   ; push (- n 1)
CALL             ; odd?'s ENTER overwrites frame_base
                 ; ... odd? executes recursively ...
                 ; RET restores pc here
; frame_base is now stale â€” use PICK for any remaining local reads
```

---

## Self-contained lambda (no formals, no captures)

A thunk: `n = 0`, `ENTER 0` is a no-op (`frame_base = data_stack.size()`).
Cleanup before `RET` is just `RET` â€” no frame slots to remove.

## Summary table

| Phase      | Instruction(s)              | Who emits  |
|------------|-----------------------------|------------|
| arg setup  | push each arg               | caller     |
| dispatch   | push handle, `CALL`         | caller     |
| prologue   | `ENTER n`, `SET_LOCAL`s     | callee     |
| body       | arbitrary, `GET_LOCAL`/`PICK` | callee   |
| epilogue   | `NROT`, `DROP`Ă—n, `RET`     | callee     |
